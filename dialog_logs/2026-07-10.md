I'm seeing a type mismatch issue in the loop where a signed integer `i` is being compared against an unsigned value `n - 1`, which triggers a sign-compare warning. The same problem appears in the nested loop condition with `j < n - i - 1`.

## Суть пункта

Проблема в **смешивании знаковых и беззнаковых целых типов**. Разберём на твоём `sort_l2.cpp:27`:

```cpp
size_t n = 10000;                        // беззнаковый (unsigned)
for (ptrdiff_t i = 0; i < n - 1; i++)    // i знаковый, n - 1 беззнаковый
```

Здесь `i` типа `ptrdiff_t` (**знаковый**, может быть отрицательным), а `n - 1` типа `size_t` (**беззнаковый**, только ≥ 0). Сравнение `i < n - 1` — это сравнение знакового с беззнаковым.

### Почему это опасно

Чтобы сравнить, компилятор приводит оба к одному типу — и приводит **знаковый к беззнаковому**. Если `i` вдруг отрицательное, оно превращается в огромное положительное число:

```cpp
ptrdiff_t i = -1;        // хотели "минус один"
size_t    u = i;         // стало 18446744073709551615 (максимум size_t)
```

В твоём цикле `i` не станет отрицательным, поэтому баг не проявится. Но компилятор под `/W4` всё равно выдаёт **предупреждение sign-compare** — а твоя политика (§ 4.1) требует зелёную сборку без предупреждений. Плюс в других файлах (`sort_l4`, `search_l5`) ты используешь `int` для размеров, а `int` на 32 бита не вместит большие коллекции.

### Что значит «тип по домену»

«Домен» = смысл величины. Размер/индекс коллекции **по своей природе неотрицателен**, поэтому его правильный тип — `std::size_t` (именно его возвращает `.size()` у контейнеров). Тогда всё сравнивается однородно, без приведений:

```cpp
size_t n = 10000;
for (size_t i = 0; i < n - 1; i++)       // оба беззнаковые — предупреждения нет
```

Ключевая мысль из ревью: **бери правильный тип с самого объявления**, а не пиши `int`, а потом кастуй `(int)v.size()` или тушуй предупреждения.

### Одна тонкость — обратные циклы

Есть ловушка: если цикл идёт **вниз** до нуля с `size_t`, он зациклится, потому что `size_t` не бывает отрицательным:

```cpp
for (size_t i = n - 1; i >= 0; i--)   // БАГ: i >= 0 всегда истина, после 0 будет huge
```

Вот для таких случаев и нужен знаковый `std::ptrdiff_t`. У тебя в `sort_l2` циклы идут **вверх**, поэтому там правильный тип — `size_t`, а `ptrdiff_t` на строке 27 просто ошибочно выбран.
