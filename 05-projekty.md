### Шаблон описания проекта

Каждая карточка содержит:

- **Что.** 1–2 предложения о сути задачи и подходе.
- **Зачем.** 1–2 предложения о практической ценности и/или для кого делается.
- **Минимальный результат.** Что должно работать к защите.
- **Стек.** Конкретные инструменты: язык, ключевые библиотеки. Здесь живёт язык программирования (Python, C++, JavaScript) и любые специфичные технологии.
- **Темы из [§0](00-temy.md).** Точные названия тем из каталога §0 через `·`. В скобках — конкретика, если важна (например: «Каркасы программных систем (Qt)», «Языки программирования (C++)»). Не-темы (парсинг, статанализ, алгоритмы хеширования) сюда не попадают — они в стеке.
- **Трудоёмкость.** *семестровая* (≈100–150 ч) / *курсовая* (семестр + защита) / *дипломная* (ВКР, 1–2 года) / *магистерская* (2 года, обычно с публикацией) / *R&D* (3+ лет, обычно в составе команды НИР).
- **См. также.** Родственные проекты в §5 — через ID (например: «См. также: П1, М16, П34»).

Авторские формулировки от наставника (унаследованы из исходного плана) сохранены как **развёрнутые описания** под полями карточки. Конкретное наполнение и сроки обсуждаются с наставником индивидуально под выбранную направленность.

##### Атласы медицинских изображений

Интерактивный анатомический атлас (2D).** Программа для просмотра анатомических срезов с навигацией, подписями, поиском, фильтром по системам органов — учебное пособие для студентов-медиков. *Минимум:* загрузка серии срезов, навигация по слайсам, текстовые подписи, поиск по названиям; *стек:* Python / JavaScript + Qt или web; *темы из §0:* Языки программирования · GUI · UI/UX; *трудоёмкость:* семестровая.

- **М3. Учебный атлас рентгенограмм с разметкой патологий.** Каталог рентгеновских снимков с интерактивными аннотациями и режимом тестирования знаний (закрытые подписи + проверка) — для подготовки врачей-рентгенологов. *Минимум:* загрузка коллекции снимков, наложение разметки, режим теста; *стек:* Python / JavaScript + GUI или web; *темы из §0:* Языки программирования · GUI · UI/UX; *трудоёмкость:* семестровая. *См. также: П30 (3D-цефалометрия — R&D-уровень анализа снимков головы).*

##### Сегментация органических структур

> R&D-уровень общей задачи — см. **П34** (Сегментация на базе нейронных сетей).

**П1. Сегментация органических структур, в первую очередь сердца** *(дипломная)*

**Что.** Автоматическое выделение анатомических структур (камеры сердца, миокард, крупные сосуды) на сериях МСКТ/МРТ с использованием нейронных сетей. Маска экспортируется в NIfTI / DICOM-RT или 3D-меш для дальнейшей обработки.

**Зачем.** Базовый шаг для последующих задач: расчёт объёмов камер, планирование операций, симуляция (см. П29 «Симулятор абляции»), парный pipeline с расчётом плана облучения (см. М10 — маска опухоли подаётся на вход дозиметрии).

- **Минимальный результат:** Dice-метрика ≥ 0.85 на отложенной выборке по основным структурам.

  *Физическая модель:* один пучок заданной ширины качается в диапазоне углов; стол с пациентом вращается относительно пучка (как в реальных ротационных установках, например в протонных ускорителях). В программной реализации это эквивалентно вращению маски (мишень + ткани) относительно неподвижного пучка.

  *Сценарий прототипа:*
  1. Загрузка DICOM МРТ-среза пациента
  2. Загрузка маски опухоли (результат бинарной сегментации, см. М16)

  4. Расчёт дозовой карты → сохранение как DICOM-срез

  *Расширения:* оптимизация параметров качания (целевая `F = покрытие_мишени − w × облучение_здоровых`, `scipy.optimize`); переход на 3D с серией срезов; неоднородное ослабление по HU.

  *См. также:* П4 (3D-версия); М16 (сегментация опухоли — вход для этого проекта).

> 6. **Docker-контейнер** для развёртывания на VPS / в облаке (Yandex Cloud, Selectel, AWS EC2 или аналоги).
> 7. UI — на выбор: CLI-утилита (минимум), нативное desktop-приложение (см. тему §0.2 «Нативные C++-приложения с WebView UI» — паттерн с Saucer / webview/webview), либо минимальная веб-форма поиска поверх REST.
>
> **Архитектурный задел:** полиморфизм по формату — таблица `files` с колонкой `format`, интерфейс `IFileIndexer` с реализацией `DicomIndexer`. Это позволяет добавить **NIfTI** (объёмные данные для нейровизуализации/сегментации), **3D-меши** (OBJ/PLY/STL/VRML — экспорт сегментированных структур) и другие SOP-классы DICOM (SR / PR / SEG / RT Structure Set) без переписывания ядра.
>
> **Расширения (в магистратуру / R&D):** DICOM-рендеринг в браузере (windowing, слайсы); метки на уровне Instance; многопользовательский режим с авторизацией; семантический поиск по эмбеддингам; ML-обучение на размеченных выборках; полная мульти-форматная индексация.

Для каждой позиции извлекается соответствующий набор контактных пар и коэффициентов согласованности. Дополнительно расскажу, тема интересная.

**П17. Оценка продуктивности и трудоёмкости разработки ПО** *(семестровая)*

> **Развёрнутое описание (наставник):**
>
> Нужно взять мои наработки по сбору статистик программистов и прикрутить туда расширенную оценку коммитов. Для этого можно попробовать использовать LLVM, делать прекомпиляцию и вычислять суммарную трудоёмкость сделанной работы. Например, при подсчёте строк добавленного кода каждой строке устанавливается балл. Конструкциям `if`, `else`, `for` ставятся наивысшие баллы. Это очень важный инструмент для компании, поскольку нужно стремиться формировать вычислительную оценку продуктивности разработчиков и меньше полагаться на субъективную оценку.

**П36. Детектор клонов кода с LLM-верификацией** *(дипломная / магистерская)*

**Что.** Многоязычный детектор клонов кода 4 типов по стандартной классификации Roy/Cordy — Type-1 (идентичные с поправкой на пробелы и комментарии), Type-2 (переименование идентификаторов/литералов), Type-3 (near-miss — мелкие вставки/удаления), Type-4 (семантические — одна задача, разный синтаксис). Type-1/2 — Merkle-хеши AST-дерева; Type-3 — Jaccard-сходство n-грамм токенов и Merkle-хешей поддеревьев с адаптивными порогами по размеру функции; Type-4 — грубый фильтр по метрикам (LOC, сложность, параметры) + LLM-верификация (OpenAI / Anthropic / Gemini / DeepSeek).

**Зачем.** Дублирование кода — основной источник technical debt и ошибок при правках. Существующие промышленные инструменты (PMD CPD, jscpd, SonarQube) находят синтаксические клоны Type-1/2/3, но Type-4 (семантические) почти не покрывают. Углублённый подход с AST-Merkle и LLM-верификацией даёт принципиально лучшее покрытие Type-4 и более точные результаты по Type-3.

- **Минимальный результат:** детектор Type-1 и Type-2 на одном языке (например, Python через `ast.parse`) с отчётом в JSON или HTML.
- **Стек:** Python 3.11+ (для `tomllib` в стандартной библиотеке); `tree-sitter` + соответствующие grammar-пакеты (`tree_sitter_javascript`, `tree_sitter_typescript` и др.) для языков, кроме Python; SDK провайдеров LLM для Type-4 (`openai`, `anthropic`, `google-genai`); HTML-генерация с интерактивностью (accept/fix кнопки в отчёте).
- **Темы из §0:** Языки программирования (Python) · Структуры данных · Регулярные выражения · Шаблоны программирования (Provider pattern для LLM-провайдеров) · Нейронные сети. LLM · Системы непрерывной интеграции кода (strict-режим для CI).
- **См. также:** §2.1 «Клоны кода» (учебная база с готовыми инструментами PMD CPD / jscpd / SonarQube); П17 «Оценка продуктивности коммитов» (родственная задача парсинга AST); П18 «Coverage-assistant» (родственный QA-инструмент).

> **Развёрнутое описание:**
>
> **Поддерживаемые языки** (через `tree-sitter` или нативные парсеры): Python, JavaScript, TypeScript, Go, Java, C#, Ruby, PHP, C, C++, Rust, Kotlin, Swift.
>
> **Пайплайн анализа:**
>
> 1. **Сбор файлов и парсинг.** Для Python — `ast.parse` (стандартная библиотека); для остальных языков — `tree-sitter` с языковыми grammar-пакетами. Извлекаются блоки кода: функции, методы, классы.
> 2. **Type-1 (идентичные).** Каждый блок получает Merkle-хеш AST-дерева — поддерево хешируется по своим узлам и хешам детей. Совпадение хешей = клон.
> 3. **Type-2 (переименование).** Перед хешированием все идентификаторы заменяются на `$VAR`, литералы на `$LIT` — нормализованный Merkle-хеш. Этот же хеш у двух разных блоков = Type-2 клон.
> 4. **Type-3 (near-miss).** Попарное сравнение блоков по двум метрикам: Jaccard-сходство n-грамм токенов и Jaccard по множествам Merkle-хешей поддеревьев. Берётся максимум. **Адаптивные пороги:** 50% для коротких функций, 55% для средних, 70% для длинных (короткие функции склонны к ложным совпадениям).
> 5. **Type-4 (семантические).** Грубый эвристический фильтр по метрикам (LOC, цикломатическая сложность, число параметров, bag-of-keywords, сходство имён) отбирает кандидатов; затем LLM-провайдер верифицирует каждую пару, отсеивая ложные.
>
> **Baseline — отслеживание клонов во времени.** При первом запуске создаётся `.clone-baseline.toml` со всеми найденными клонами. При последующих запусках детектор сравнивает текущий скан с baseline и выводит **только новые** клоны. Через интерактивный HTML-отчёт пользователь помечает клоны как `accepted` (допустимые) или `fix` (требуют исправления) — помеченные не генерируют предупреждений впредь. Strict-режим возвращает exit code 1 в CI при появлении новых клонов.
>
> **Форматы вывода:** текстовый, JSON (для машинной обработки), HTML (с интерактивной разметкой).
>
> **Расширения (в магистратуру):** сравнение качества с PMD CPD / jscpd / SonarQube на эталонных корпусах (BigCloneBench и др.); метрики precision/recall по типам клонов; публикация результатов; интеграция с pre-commit hooks; собственное обучение классификатора Type-4 как замена LLM (для офлайн-использования и снижения стоимости).

**П37. Анализ покрытия тестов в Visual Studio (Microsoft.CodeCoverage.Console + diff-coverage + кратность hits)** *(курсовая / дипломная)*

**Что.** Учебный полигон для исследования встроенных средств анализа покрытия кода в Visual Studio 2026 для нативного C++. Включает запуск через `Microsoft.CodeCoverage.Console` с динамическим инструментированием, подбор оптимальных флагов компиляции (`/O2 /Ob1` — баланс оптимизации и информативности отчёта), накопление кратности покрытия через ReportGenerator merge, генерацию Cobertura XML только для изменённого кода (diff-coverage с транзитивным анализом по графу вызовов из Doxygen XML), и собственный HTML-отчёт с визуализацией кратности (цветовой градиент по hits, гистограмма распределения).

**Зачем.** Покрытие — ключевая метрика качества тестов, но Microsoft Code Coverage из коробки даёт только бинарный признак «покрыто / не покрыто» (hits = 0 или 1). ReportGenerator merge даёт **кратность hits** — это позволяет находить «слабые» строки (покрыты только одним тестом, риск необнаруженной регрессии при поломке этого теста) и избыточное тестирование (несколько тестов на одну и ту же строку — кандидаты на оптимизацию). **Diff-coverage** показывает покрытие именно изменённых строк PR — критично для CI на больших проектах, где гонять полное покрытие на каждый PR непрактично.

- **Минимальный результат:** запуск с одним тестом, генерация Cobertura XML, базовый HTML-отчёт, diff-coverage по последнему коммиту.
- **Стек:** Visual Studio 2026 (Community / Professional / Enterprise — теперь все поддерживают C++ coverage; раньше только Enterprise) + `Microsoft.CodeCoverage.Console` + ReportGenerator (`dotnet tool install -g dotnet-reportgenerator-globaltool`) + Python 3.x (для собственных скриптов diff/html) + Doxygen (для транзитивного анализа графа вызовов) + Cobertura XML (стандартный формат, совместим с Azure DevOps, GitHub Actions, GitLab CI, SonarQube) + git (для diff).
- **Темы из §0:** Языки программирования (C++) · Совместная разработка. Системы контроля версий (Git) · Модульное тестирование · Метрики тестирования · Системы непрерывной интеграции кода · Языки разметки (XML / HTML) · Генерация программной документации (Doxygen для графа вызовов).
- **См. также:** П18 «Coverage-assistant» (родственный QA-инструмент — общий концепт привязки тестов к коммитам); §2.1 «Метрики покрытия кода» (учебная база — gcov, OpenCppCoverage, MS Code Coverage).

> **Развёрнутое описание:**
>
> **Что нового в Visual Studio 2026:**
>
> - `Microsoft.CodeCoverage.Console` — новый инструмент командной строки, заменяет старые `vsinstr` / `vsperfcmd`.
> - **Динамическое инструментирование** — не нужно отдельно инструментировать exe перед запуском.
> - **Покрытие C++ в Community/Professional** — раньше было только в Enterprise.
> - Поддержка x86, x64, ARM64.
>
> **Подбор флагов компиляции для информативного покрытия:**
>
> | Флаги | Строк в отчёте | Комментарий |
> |---|---|---|
> | `/O2` | 13 | Агрессивный инлайнинг, мало информации |
> | `/O2 /Ob1` | **39** | **Оптимальный баланс** — макс. оптимизация с отключённым автоинлайнингом |
> | `/O2 /Ob0` | 37 | Хорошо, но чуть меньше оптимизации |
> | `/O1` | 16 | Много инлайнинга |
> | `/Od` | 41 | Максимум информации, но без оптимизации |

> **Накопление кратности hits** через ReportGenerator merge: запустить каждый тест отдельно (создаются отдельные `.coverage` файлы) → объединить через `reportgenerator -reports:*.coverage -targetdir:report` → суммируются hits, и в отчёте видно, сколько разных тестов покрывают каждую строку.
>
> **diff_cobertura.py** — генератор Cobertura XML только для изменённых строк (по `git diff` относительно ветки/коммита/диапазона). Опционально — транзитивный анализ через Doxygen XML: добавляются строки функций, которые **вызывают** изменённый код (через граф callers) — это позволяет оценить, покрыты ли тестами места использования изменённой функции.
>
> **html_report.py** — собственный HTML-генератор как альтернатива ReportGenerator (без рекламы PRO-версии) и OpenCppCoverage (без поддержки кратности): круговая диаграмма общего покрытия, гистограмма распределения hits, таблица файлов, подсветка исходного кода с цветовым градиентом по кратности (красный = 0, светло-зелёный = 1, тёмно-зелёный = много), badge с hits на каждой строке.
>
> **Расширения (в магистратуру):** интеграция в CI с гейтом по diff-coverage (PR не проходит, если новые строки покрыты ниже X%); сравнение покрытия двух разных тестовых стратегий по гистограмме hits; собственная метрика «эффективность теста» = (новое покрытие, добавленное тестом) / (время выполнения теста).

**П38. Cache-friendly in-memory структуры данных: B-tree vs RB-tree (с автономным агентным harness)** *(семестровая → магистерская, четыре уровня)*

**Что.** Эмпирически показать, что B-tree обгоняет красно-чёрное дерево на больших in-memory коллекциях не за счёт асимптотической сложности (она у обоих O(log n)), а за счёт меньшей высоты и меньшего числа cache miss'ов. Реализовать обе структуры, измерить, объяснить разницу. На старших уровнях — добавить B+-tree, SIMD-поиск, сравнение с hash map / ART, и завершить **autoresearch-harness**, в котором AI-агент сам автономно оптимизирует cache-aware B-tree через цикл «гипотеза → эксперимент → метрика → принять/откатить» (по схеме [karpathy/autoresearch](https://github.com/karpathy/autoresearch)).

**Зачем.** Понять связку «высота дерева ↔ размер узла ↔ cache locality» — основу всех современных in-memory индексов СУБД (PostgreSQL, MySQL InnoDB, SQLite) и cache-conscious структур данных. Это классический сюжет «O-нотация лжёт»: асимптотика одинаковая, а на больших данных разница в 5–10×. Уровень 4 даёт навык, актуальный для индустрии 2026+: проектировать **eval harness, в котором AI-агент делает работу инженера, а человек становится научным руководителем**.

- **Минимальный результат для защиты (любой уровень):** работающая реализация хотя бы одного варианта B-tree + численное сравнение с RB-tree на 1M элементов + объяснение причин различий со ссылкой на cache miss как первопричину.
- **Стек:** C++ (для прямого контроля над памятью и SIMD); Python для построения графиков (`matplotlib`); `perf` (Linux) или Intel VTune / AMD μProf (Windows) для измерения cache misses; CMake + Google Benchmark; на уровне 4 — Claude Code / Aider / OpenCode для агента.
- **Темы из §0:** Языки программирования (C++) · Структуры данных · Профилирование кода · Метрики тестирования · Распределённые и параллельные вычисления (для SIMD на уровне 3) · Работа с AI-агентами разработки (на уровне 4) · Шаблоны программирования (eval harness как паттерн) · Системы непрерывной интеграции кода (для воспроизводимости).
- **См. также:** §2.1 «Структуры данных» (учебная база); §0.2 «Работа с AI-агентами разработки» (autoresearch-паттерн на уровне 4); П20 «Параллельный решатель СЛАУ» (родственная тема — высокопроизводительные вычисления с памятью).

> **Развёрнутое описание:**
>
> ### Уровень 1 — семестровая (минимум)
>
> 1. Реализовать B-tree порядка K (узел = массив ключей фиксированного размера, бинарный поиск внутри узла, split при переполнении).
> 2. Загрузить 1M случайных `int`-ключей.
> 3. Сравнить среднее время 100K случайных exact-match поисков для:
>    - `std::set<int>` (стандартное RB-tree);
>    - своё B-tree порядков 16 / 64 / 256;
>    - `std::unordered_map<int, int>` (hash map — для контрольной точки).
> 4. Построить таблицу «структура → время поиска» и объяснить, почему RB-tree медленнее, почему B-tree порядка 64 быстрее порядка 16, и почему hash map обычно ещё быстрее на чистом exact-match.
>
> ### Уровень 2 — курсовая
>
> 1. Добавить **B+-tree**: все значения в листьях, листья связаны в linked list для range-scan.
> 2. Реализовать `range_scan(a, b) → list<int>` и сравнить производительность с `std::set::lower_bound + iter` и hash map (где это сделать сложнее).
> 3. **Измерить cache misses**: `perf stat -e cache-misses,L1-dcache-load-misses` (Linux) или Intel VTune Memory Access (Windows). Воспроизвести таблицу из источника на Habr — получится 6.7× ускорение или больше/меньше, и почему?
> 4. Зафиксировать конфигурацию железа: CPU, размеры L1/L2/L3, аллокатор (`malloc` vs pool/slab), флаги компиляции (`-O2`, `-march=native`).
>
> ### Уровень 3 — дипломная
>
> 1. Реализовать оптимизации:
>    - **SIMD-поиск внутри узла** (AVX2 для x86 / NEON для ARM) — 2–3× ускорение поиска внутри узла.
>    - **Bulk loading** с предсортировкой массива ключей — построение дерева в 10–100× быстрее, чем последовательная вставка.
>    - **Embedded-вариант** с массивом узлов и 16-битными индексами вместо указателей (экономия 6 байт на ссылку, нет вызовов `malloc`).
> 2. Сравнить с **Adaptive Radix Tree (ART)** — взять стороннюю библиотеку (`libart`) или реализовать свой урезанный вариант.
> 3. Опубликовать **воспроизводимый бенчмарк**: CMake-проект + Docker для одинакового окружения + скрипт прогона + графики через `matplotlib` + сырые `perf`-отчёты.
>
> ### Уровень 4 — магистерская / R&D (autoresearch-паттерн)
>
> На уровнях 1–3 студент пишет код сам и оптимизирует руками. На уровне 4 студент **меняет роль**: вместо инженера становится **руководителем эксперимента**. AI-агент перебирает варианты, студент проектирует harness, фиксирует метрику, контролирует честность результатов и пишет нарратив.
>
> **Архитектура harness** (по аналогии с karpathy/autoresearch — `prepare.py` / `train.py` / `program.md`):
>
> | Файл | Роль | Кто редактирует |
> |---|---|---|
> | `prepare.py` | Зафиксированные константы: датасет 1M ключей с заданным распределением (uniform / zipf / sequential), бенчмарк-набор из 100K запросов, выбор аллокатора, hidden test set для cheat-detection | Студент один раз, агент не трогает |
> | `bench.cpp` | Скелет B-tree + бенчмарк. Метрика — `cycles_per_lookup` (через `__rdtsc`) и `cache_misses_per_lookup` (через `perf_event_open`). Запуск ровно 30 секунд, N итераций для стабилизации | Агент модифицирует |
> | `run.py` | Запускалка: компилирует `bench.cpp` (`-O3 -march=native`), прогоняет, парсит вывод, сравнивает с `best.json`, **принимает или откатывает** изменение | Зафиксировано |
> | `program.md` | Инструкции агенту: что можно/нельзя менять, какие идеи стоит попробовать, ограничения | Студент-руководитель |
> | `experiments.log` | Журнал прогонов: дата, гипотеза, изменение, метрика, объяснение | Автозапись |
> | `best.json` | Текущий чемпион | Автозапись |
>
> **Цикл агента** (повторяется автономно, ~30–60 с на итерацию, 500+ итераций за ночь):
>
> 1. Прочитай `program.md` и `experiments.log` (что уже пробовали).
> 2. Сформулируй гипотезу для следующего изменения. Запиши.
> 3. Модифицируй `bench.cpp`.
> 4. Вызови `run.py`.
> 5. Если новая метрика лучше best — обнови `best.json` и зафиксируй. Хуже — откати `bench.cpp`.
> 6. Запиши в `experiments.log`: гипотеза, изменение, результат, объяснение.
> 7. К п.1.
>
> **Пространство поиска** (студент перечисляет в `program.md`):
>
> - Порядок узла (4, 8, 16, 32, 64, 128, 256, ...).
> - Layout узла (массив структур vs параллельные массивы ключей и детей).
> - Внутри-узловой поиск (линейный / бинарный / SIMD AVX2 / SIMD NEON).
> - Branch hints (`__builtin_expect` / `[[likely]]` / `[[unlikely]]`).
> - Software prefetching (`__builtin_prefetch` для следующего узла во время сравнения текущего).
> - Padding и alignment (`alignas(64)` под cache line).
> - Sentinel-ключи для упрощения условий выхода.
> - Pointer compression (16/32-bit индексы вместо 64-bit указателей).
>
> **Запрещено агенту:** удалять `assert`'ы, менять `prepare.py` / `run.py` / метрику / бенчмарк, делать compile-time трюки на конкретных ключах бенчмарка.
>
> **Cheat-detection** (важнейшая часть harness):
>
> - **Две метрики обязательны.** `cycles_per_lookup` (что оптимизируем) + `correctness_check` (после каждого изменения прогон на эталонной выборке, сверка с `std::set`). Расхождение = автоматический откат.
> - **Hidden test set** в `prepare.py` — ключи, которых нет в публичном бенчмарке. Прогоняются после каждого принятого изменения. Если падают — откат и красная метка.
> - **Code review агентом № 2** — отдельный проход «code reviewer agent», который читает diff и помечает подозрительные паттерны: `static unordered_map`-кэши, `if (bench_seed == ...)`, закомментированные `assert`'ы, ранние `return` для известных ключей.
>
> **Что делает студент:**
>
> 1. **Проектирует harness** (1–2 недели) — формат `experiments.log`, формат `program.md`, cheat-detection. Это самая сложная часть; агент не должен иметь возможности «обмануть» harness.
> 2. **Запускает агента на ночь** — 500+ экспериментов. Утром смотрит `experiments.log`.
> 3. **Анализирует результаты** — какие изменения дали ускорение, какие нет, какие регрессировали. Группирует находки по типам (структурные / микрооптимизации / SIMD).
> 4. **Объясняет находки в терминах теории** — почему `prefetch` работает, почему SIMD-поиск перестаёт помогать на узлах <16 ключей, какой baseline лучше всех. Это и есть «грок» предметной области.
> 5. **Сравнивает с литературой:** совпадает ли с Rao & Ross 2000? С Bender et al. 2005? Что агент **упустил**, хотя мог бы найти? Что **открыл нового** (или думает, что нашёл, а на самом деле уже опубликовано)?
> 6. **Пишет защитный нарратив:** лучший вариант агента + объяснение + сравнение с RB-tree и hash map + ограничения подхода + 3 идеи как улучшить harness.
>
> **Минимальный результат уровня 4:** работающий harness, реально автономно прогоняющий ≥ 50 итераций без вмешательства; журнал с осмысленными комментариями; доказательство, что cheat-detection ловит специально подсунутый cheat (для проверки harness); финальный отчёт со сравнением «лучший агентский результат vs ручная реализация уровня 1–3 vs литературный baseline».
>
> ### Магистерское / R&D расширение уровня 4
>
> - **Multi-objective optimization:** оптимизировать одновременно `cycles_per_lookup`, размер бинарника, потребление памяти. Pareto-front, не одна точка.
> - **Бенчмарк агентов:** тот же harness с разными агентами (Claude Sonnet 4.6 vs Opus 4.7 vs DeepSeek vs Gemini) — кто эффективнее находит оптимизации? Это и есть **бенчмарк агентов на инженерной задаче** (а не на синтетических MMLU/HumanEval).
> - **Meta-prompting:** второй агент анализирует `experiments.log` и переписывает `program.md`, чтобы ускорить сходимость. Двухуровневый цикл (внутренний — оптимизация B-tree, внешний — оптимизация инструкции).
>
> ### Подводные камни (для честной оценки на любом уровне)
>
> - **Цифры из источников иллюстративные.** Без указания CPU, кэша, аллокатора цитируемые числа (например, 6.7× ускорение из Habr-главы) нельзя цитировать как факт. На вашем железе результат может быть 2× или 15×.
> - **Hash map в exact-match почти всегда обгоняет B-tree.** Включайте его в свою таблицу — это показывает интеллектуальную честность.
> - **«Узел = одна cache line»** — терминологическая ошибка многих учебников. Узел B-tree порядка 64 для int-ключей ≈ 1.3 КБ ≈ 20 cache lines. Работает hardware prefetcher, который последовательно подгружает линии после первого miss. Это стоит проверить эмпирически (`L1-dcache-prefetches`).
> - **На современных x86 с многомегабайтным L3 эффект меньше**, чем на ARM Cortex-M с 64 КБ кэша.

**П39. Учебный CI/CD pipeline для C++ проекта** *(курсовая / дипломная)*

**Что.** Построить с нуля автоматический pipeline сборки и доставки для нативного C++ проекта на GitHub Actions: матричная сборка под несколько ОС/компиляторов → линтер и форматтер → юнит-тесты с покрытием → упаковка артефакта (zip с бинарём + sha256) → автоматический релиз по тегу. В роли «прикладного» проекта берётся один из учебных CLI студента (например, П21 DuplicateFileFinder или любая из задач, на которой нарабатывалась §2.1).

**Зачем.** В §0.2 CI/CD упомянут как тема, но без практического полигона остаётся «знаю, что есть GitHub Actions». В реальности первая собственная сборка обвязки CI/CD занимает у студента-новичка несколько недель: матрицы, секреты, кэш зависимостей, разница между push и PR-триггерами, правильная упаковка артефактов, создание релиза по тегу — всё это узнаётся только руками. Этого практического опыта явно не хватало предыдущим студентам — DevOps-навыки приходилось добирать уже после трудоустройства. Проект закрывает эту дыру и дает студенту артефакт «зелёный CI на моём C++ проекте», который хорошо смотрится в портфолио на собеседовании.

- **Минимальный результат:** GitHub Actions workflow собирает один CLI на 2 ОС (Windows + Linux) под `cl` и `gcc`/`clang` соответственно; запускает `clang-format --dry-run -Werror` и `clang-tidy`; прогоняет GoogleTest через `ctest --output-on-failure`; собирает артефакт через CPack или скрипт упаковки и прикладывает его к workflow run; по push-у тега `v*.*.*` создаёт GitHub Release с приложенным архивом.
- **Стек:** GitHub Actions (YAML); CMake + CTest + GoogleTest; `clang-format`, `clang-tidy`; CPack или Python-скрипт упаковки; опционально — OpenCppCoverage / gcov + публикация отчёта о покрытии в артефакты; опционально — Docker-образ как альтернативная цель сборки.
- **Темы из §0:** Языки программирования (C++) · Совместная разработка. Системы контроля версий (Git) · Модульное тестирование · Системы непрерывной интеграции кода · CI/CD расширенный · Развёртывание и обновление приложений · Контейнеризация и Docker (опционально).
- **См. также:** §2.1 «CI/CD, артефакты и упаковка» (учебная база); П18 «Coverage-assistant» и П37 «Анализ покрытия в VS» (родственные QA-инструменты, естественно встраиваются в pipeline); П21 «DuplicateFileFinder» как типичный «прикладной» проект внутри pipeline.

> **Развёрнутое описание:**
>
> **Этапы выполнения (минимальная конфигурация):**
>
> 1. **Базовая сборка.** Один workflow `.github/workflows/build.yml`, триггеры `push` и `pull_request`, одна job `build`, одна ОС (`ubuntu-latest`), шаги `checkout` → `cmake -B build` → `cmake --build build` → `ctest --test-dir build --output-on-failure`. Зелёный значок в README.
> 2. **Матрица.** Превратить job в матрицу `os: [ubuntu-latest, windows-latest]` и `compiler: [gcc, clang, msvc]` с правильными `if`-ограничениями (msvc только на windows и т.п.). Понять, как матрица генерирует Cartesian product.
> 3. **Линтер и форматтер.** Отдельная job `lint`: `clang-format --dry-run -Werror` (проверка стиля) + `clang-tidy` через `run-clang-tidy.py`. Job `lint` должна блокировать мерж не хуже, чем `build`.
> 4. **Кэш зависимостей.** `actions/cache` для CMake-кэша / vcpkg / `~/.cache/clang-tidy`. Замер ускорения сборки до/после.
> 5. **Артефакты.** `actions/upload-artifact` с собранным бинарём + sha256-чек-суммой. Скачать из workflow run и проверить запуск.
> 6. **Релиз по тегу.** Отдельная job `release` с триггером `on: push: tags: ['v*.*.*']`, `softprops/action-gh-release` или `gh release create`, прикладывает архив с бинарями под обе ОС. Создать тег `v0.1.0` и проверить, что Release появляется автоматически.
>
> **Расширения для дипломного уровня:**
>
> - **Покрытие в CI.** Сборка с `--coverage` (gcc) / OpenCppCoverage (Windows) → парсинг отчёта → публикация в артефакты + опциональная интеграция с Codecov / SonarCloud. Гейт «PR не мержится, если diff-coverage < 80%».
> - **Санитайзеры.** Отдельная job под ASan / UBSan / TSan — раз в день / на push в `main` (не на каждый PR — медленно).
> - **Pre-commit hooks локально.** `pre-commit` framework с теми же `clang-format` / `clang-tidy` — гарантия, что локальный коммит проходит CI с первой попытки.
> - **Контейнерная сборка.** Dockerfile с фиксированной версией компилятора + GitHub Actions job, использующий этот образ. Воспроизводимость окружения сборки.
> - **Кросс-компиляция.** Сборка под ARM64 через QEMU или `cross-rs`-подобную обёртку.
> - **Подписание артефактов.** Sigstore / cosign — современный стандарт supply chain security.
>
> **Что считается успехом на защите:** живой репозиторий с зелёным CI, демонстрация push в PR (видно, как линтер и тесты гоняются параллельно), демонстрация создания релиза по тегу (видно, как готовый архив появляется в Releases), и устный разбор: что именно делает каждая job, почему она нужна, что делать, когда CI падает, как читать логи.
>
> **Подводные камни (обсудить с наставником):**
>
> - GitHub Actions free-tier лимиты на private-репо — для учебного проекта используем публичный.
> - Кросс-платформенные различия в путях, переносах строк, командах оболочки (`bash` vs `pwsh`) — повод понять, что значит «инфраструктура — это код».
> - Время сборки: новичок легко сделает CI на 30 минут на каждый PR — лимиты бесплатного тира забьются. Кэш и параллельность — обязательная часть, не «опциональная».
> - Секреты (например, токен для деплоя или подписи) — в репозитории не светим, используем `secrets.GITHUB_TOKEN` и Encrypted Secrets.

**П40. Применение статанализаторов к учебной кодовой базе** *(семестровая)*

**Что.** Прогнать несколько статических анализаторов кода на одной C++-кодовой базе (один из учебных CLI студента из §2.1, либо небольшой open source проект средней зрелости), составить сводный отчёт по найденным предупреждениям и выработать дисциплину разбора выхлопа: какие предупреждения — true positive, какие — false positive (и почему), какие пересекаются между анализаторами, какие уникальны для одного из них.

**Зачем.** «Запустить инструмент» — это 5% работы; основное время уходит на **разбор результата**. Для хорошего результата нужно понимать, что инструмент видит, что не видит, и какой класс багов какой инструмент ловит лучше других. Этот навык не даётся ни в одной отдельной теме — нарабатывается только на практике сравнения нескольких инструментов на одном коде. Дополнительно проект закрывает дыру, явно проявившуюся у предыдущих студентов: невнимательность при коммитах и стилевые мелочи (табуляции/пробелы, забытый `const`, мусор в коммитах) системно гасятся не устными замечаниями наставника, а статанализом и линтером в pre-commit/CI — этому проекту учится по дороге.

- **Минимальный результат:** прогон **двух open source анализаторов** (`clang-tidy` + `cppcheck`) на учебной кодовой базе; сводный отчёт минимум по 30 предупреждениям с разметкой каждого как «true positive / false positive / стилевое / опечатка инструмента»; интеграция обоих анализаторов в CI как блокирующая job (`exit code 1` при наличии новых предупреждений).
- **Стек:** C++ кодовая база (свой проект из §2.1 — например, обвязка вокруг полигонов или DuplicateFileFinder — либо опенсорсный референс); `clang-tidy` (LLVM, open source), `cppcheck` (GPLv3, open source); CMake-интеграция (`CMAKE_CXX_CLANG_TIDY`, скрипт-вызов cppcheck); GitHub Actions для CI; Markdown/HTML генератор отчёта (Python или Pandoc).
- **Темы из §0:** Языки программирования (C++) · Стандарты кодирования (форматирование) · Статический анализ кода · Системы непрерывной интеграции кода · Совместная разработка. Системы контроля версий (Git).
- **См. также:** §2.1 «Форматирование (стиль)» и «Статический анализ корректности» (учебная база); П39 «Учебный CI/CD pipeline» (статанализ — естественная job в pipeline); П36 «Детектор клонов кода» (родственная R&D-задача — создание собственного анализатора, а не применение существующих).

> **Развёрнутое описание:**
>
> **Этапы выполнения (минимальная конфигурация):**
>
> 1. **Выбор кодовой базы.** Свой учебный CLI из §2.1 после нескольких слоёв инфраструктуры — оптимальный вариант (студент уже знает код). Альтернатива — небольшой публичный C++ проект (1k–10k LOC), для которого ещё нет настроенного статанализа.
> 2. **Запуск open source инструментов вручную.**
>    - `clang-tidy --checks='*,-fuchsia-*,-llvm-header-*' file.cpp -- -std=c++20 -I include` — пройтись по всем файлам, начать с широкого набора чеков, затем сузить до релевантных.
>    - `cppcheck --enable=all --inconclusive --xml file.cpp 2> cppcheck.xml`.
> 3. **Сводная таблица.** В Markdown или CSV: один файл/строка кода, по столбцу на каждый анализатор, ячейка — короткое имя предупреждения (`bugprone-narrowing-conversions`, `nullPointer` и т. п.). Видно пересечения и уникальные находки.
> 4. **Разметка findings.** Каждое предупреждение отметить: `TP` (true positive — реальный дефект, нужно править), `FP` (false positive — инструмент ошибся), `STYLE` (вопрос стиля, не корректности), `KNOWN` (известное и допустимое — `// NOLINT(check-name)` с обоснованием в комментарии).
> 5. **Интеграция в CI.** GitHub Actions job, которая прогоняет оба анализатора и падает, если в diff появились новые предупреждения, не помеченные как `KNOWN`.
> 6. **README раздела «Качество кода».** Описать: какие анализаторы используются, какие чеки выключены и почему, как добавить `// NOLINT` правильно (с указанием check'а и обоснованием), как локально воспроизвести то же, что гоняется в CI.
>
> **Расширения для дипломного уровня (необязательно для семестровой):**
>
> - **Третий и четвёртый анализатор.** `include-what-you-use` (полезен на типичной C++-кодовой базе с разросшимися `#include`); Semgrep (паттерны на синтаксис, можно писать свои правила за 5 минут).
> - **Коммерческий анализатор по бесплатной лицензии.** **PVS-Studio** в свободном режиме для open source / индивидуального некоммерческого использования (через специальный маркер-комментарий в исходниках — см. документацию PVS-Studio). Сравнить классы ошибок: что нашёл PVS-Studio, чего не нашли open source инструменты, и наоборот. **Если действующая лицензия недоступна — вместо PVS-Studio добавить ещё один open source инструмент (Infer, Semgrep).**
> - **SonarQube как платформа.** **SonarQube Community Edition** локально (Docker-контейнер) или **SonarCloud** для public-репозитория — агрегация всех findings в одном UI с трендами по времени. Опять же — только если есть рабочий доступ.
> - **Статистика по проекту.** Метрики плотности предупшеждений (warnings per kLOC), доля FP по каждому анализатору на этой кодовой базе, время на разбор одного предупреждения. Эти цифры — материал для отчёта.
> - **Свои правила.** Написать одно правило для clang-tidy (через AST matchers) или для Semgrep (через YAML pattern) — поймать типичный для своей кодовой базы паттерн, который существующие чеки не ловят.
>
> **Что считается успехом на защите:** живой репозиторий с настроенной CI-интеграцией двух анализаторов; сводный отчёт по 30+ findings с разметкой; устный разбор: «вот это false positive, потому что инструмент не учитывает X», «вот это true positive, который никогда не выстрелил бы в тестах»; понимание, **какой класс багов какой инструмент ловит** (например: cppcheck лучше на legacy C-стиле, clang-tidy — на современных идиомах, IWYU — только про includes).
>
> **Подводные камни (для честной оценки):**
>
> - **«Зелёный анализ» с первого прогона недостижим** на чужом коде. Будут сотни предупреждений, и большая часть — стилевые / known-issue. Задача — отфильтровать сигнал.
> - **False positive ≠ «инструмент плохой».** FP возникают потому, что статанализатор не выполняет код и не знает контекста (например, `if (x) { … } else { … } // x точно был ненулевым из инварианта класса`). Это нормальная плата за раннее обнаружение.
> - **Бесплатный режим коммерческих инструментов меняется.** Перед включением PVS-Studio / SonarCloud в проект — проверить актуальные условия лицензии на их сайте. Если условия для конкретной ситуации не выполняются — заменить на open source аналог.
> - **PVS-Studio не на всех платформах ставится одинаково просто** (на Linux — пакет, на Windows — установщик; в свободном режиме нужен корректный маркер в исходниках, без него анализатор не запустится).

**П41. Графовый extension к SQLite (custom functions + virtual table)** *(курсовая → дипломная, два уровня)*

**Что.** Реализовать SQLite extension (`.dll`/`.so`), добавляющий графовые операции к обычной реляционной БД: алгоритмы поиска путей (BFS, Dijkstra, A*) над графом, хранящимся в SQLite, доступные через custom-функции и virtual table. Уровень 1 (курсовая) — pure-SQL подход через recursive CTE + первая custom scalar-function на C++. Уровень 2 (дипломная) — полный virtual-table API с C++-реализованными алгоритмами под капотом.

**Зачем.** SQL плохо подходит для графовых обходов: нет priority queue, A* в чистом SQL невозможен, recursive CTE на больших графах разваливается без явного управления планом. Это и есть причина существования отдельной категории — graph DBs (Neo4j, Apache AGE для PostgreSQL, SQL Server Graph Tables с 2017). Custom extensions позволяют добавить недостающую функциональность без перехода на отдельную графовую СУБД и сохранить C++-алгоритмы под удобным SQL-фасадом. Тот же extension-механизм лежит в основе FTS5 (полнотекст) и R*Tree (геоиндекс) — core-расширений самого SQLite. Production-grade готовых графовых extensions для SQLite **практически нет** (поиск выдаёт несколько proof-of-concept-проектов с recursive CTE patterns) — это ниша, в которой собственный артефакт может быть реально полезен и опубликован.

- **Минимальный результат для защиты:** работающий extension с хотя бы одной custom-функцией поиска пути (BFS или Dijkstra) на adjacency-list таблице; CTest-воспроизводимые тесты на корректность; README с примером загрузки extension и SQL-запросов; benchmark расширения против recursive-CTE на одинаковом графе.
- **Стек:** C++ для алгоритмической части + C для границы с SQLite API (официальное API на C); SQLite через amalgamation или vcpkg; CMake + GoogleTest + ctest; `google-benchmark` или `std::chrono` для замеров. На Windows — MSVC, на Linux — gcc/clang.
- **Темы из §0:** Языки программирования (C++, C) · Структуры данных (графы) · Введение в БД на примере SQLite · Архитектура программного обеспечения (plugin/extension-pattern) · Алгоритмы (BFS, Dijkstra, A*) · Метрики тестирования (на уровне 2).
- **См. также:** §2.1 «Структуры данных» и «Тестирование» (учебная база); §0.1 «Введение в БД» и §0.2 «Базы данных продвинутые» (теоретический контекст); П5 «Индексирование DICOM» (родственное применение SQLite в production-задаче); П38 «Cache-friendly структуры данных» (родственная техническая задача с эмпирическим сравнением алгоритмов и замерами производительности).

> **Развёрнутое описание:**
>
> ### Уровень 1 — курсовая (минимум)
>
> 1. Подключить SQLite через vcpkg или скомпилировать `sqlite-amalgamation` с `-DSQLITE_ENABLE_LOAD_EXTENSION`.
> 2. Спроектировать схему графа: `nodes(id INTEGER PRIMARY KEY, label TEXT)`, `edges(src INTEGER, dst INTEGER, weight REAL, FOREIGN KEY (src) REFERENCES nodes(id), FOREIGN KEY (dst) REFERENCES nodes(id))`. Индекс на `edges(src)` обязателен.
> 3. Программа на C++ (хост-программа, не extension): открывает .db, добавляет/удаляет узлы и рёбра через prepared statements; импортирует сериализованный граф из CSV или GraphML.
> 4. Реализовать поиск пути через **recursive CTE** (`WITH RECURSIVE paths AS (...)`) — BFS-style. Запустить на графах разного размера, **зафиксировать ограничения**: нет приоритета веса (Dijkstra нельзя), не работает A* с эвристикой, тормозит на графах > 10K вершин без `INDEXED BY` и явных подсказок планировщику.
> 5. Реализовать первый собственный extension: один файл `graphext.cpp`, entry point `sqlite3_graphext_init` с `extern "C"`, регистрирующий простую scalar-function `graph_version()` через `sqlite3_create_function`. Загрузить через `.load ./graphext` (Linux) / `.load graphext.dll` (Windows) или через `sqlite3_load_extension` из C++. Это знакомство с extension API и тонкостями границы C/C++.
> 6. Реализовать `shortest_path_dijkstra(src, dst)` как custom scalar-function. Внутри — Dijkstra на in-memory adjacency-list, прочитанной из той же базы через подготовленные `SELECT ... FROM edges`. Возврат — JSON-строка с массивом ID вершин или comma-separated text.
> 7. **Тесты на корректность** (GoogleTest + ctest): задать известный граф, проверить путь по эталону для нескольких пар src/dst; покрыть corner-cases — несвязный граф, нулевой путь (`src == dst`), несуществующие узлы, отрицательные веса (Dijkstra не должен вызываться, ошибка через `sqlite3_result_error`).
>
> ### Уровень 2 — дипломная
>
> 1. **Virtual Table API.** Перенести Dijkstra в virtual table:
>
>    ```sql
>    CREATE VIRTUAL TABLE shortest_paths USING graph_paths;
>    SELECT step_index, node_id, cumulative_weight
>    FROM shortest_paths
>    WHERE src = 'A' AND dst = 'B' AND algorithm = 'dijkstra';
>    ```
>
>    Реализовать обязательные callbacks: `xCreate`/`xConnect` (создание VTab при `CREATE VIRTUAL TABLE`), **`xBestIndex`** (планировщик: какие `WHERE`-условия передаются как параметры алгоритма — критичный для производительности), `xFilter` (выполнение запроса), `xNext`/`xColumn`/`xEof` (итерация результата), `xDestroy`/`xDisconnect` (очистка). Документация — официальный SQLite VTab API; примеры — исходники FTS5 и R*Tree в core SQLite.
> 2. **A* с эвристикой.** Добавить вторую стратегию: `WHERE algorithm = 'astar' AND heuristic = 'manhattan'`. Эвристика берётся либо из координат вершин (`nodes.x, nodes.y`), либо из предвычисленной таблицы `node_coords`. Сравнить количество посещённых вершин Dijkstra vs A* на сетках разной сложности — на сетке с препятствиями A* должен посещать в разы меньше вершин.
> 3. **Performance benchmarking.** Полная таблица замеров на синтетических графах разного типа:
>    - Случайный плоский (Erdős–Rényi) с разной плотностью.
>    - Сетка 2D с препятствиями (для A*).
>    - Дорожная сеть (взять открытый dataset OpenStreetMap subset).
>
>    Сравнить: pure recursive CTE vs custom scalar-function vs virtual table vs ручная программа на Boost Graph Library. Зафиксировать конфигурацию железа, размер графов, флаги компиляции.
> 4. **Документация и пример использования.** README с примерами: «как загрузить граф из CSV в SQLite, как выполнить поиск через VTab, что писать в `xBestIndex` для оптимизации». Сгенерировать Doxygen-документацию extension API (`@brief` на каждой публичной функции).
> 5. **Cross-platform CI.** GitHub Actions или GitLab CI: сборка extension на Windows (MSVC), Linux (gcc/clang), macOS (clang), прогон ctest на всех платформах. Артефакт релиза — `.dll`/`.so`/`.dylib` + примеры.
>
> ### Магистерское / R&D расширение
>
> - **Cypher-подобный DSL поверх extension.** Парсер мини-языка `MATCH (a)-[*..5]->(b) RETURN path` → план запроса → вызовы Dijkstra/BFS на VTab. Самостоятельная исследовательская работа на стыке language design + graph algorithms.
> - **Streaming graph updates.** Поддержка graph-триггеров — пересчёт shortest path при изменении edges (incremental Dijkstra).
> - **Concurrent access.** Анализ блокировок SQLite (WAL mode), thread-safety extension, бенчмарк параллельных запросов.
> - **GraphQL-фронтенд** к SQLite extension через простой HTTP-сервер на Boost.Beast или cpp-httplib. Превращает локальную SQLite-базу в graph-сервис.
> - **Сравнение с Apache AGE** (Postgres + Cypher): реализовать одинаковый набор запросов на обоих, замерить производительность, написать сравнительный отчёт. Это потенциальная статья на Habr или в IEEE.
>
> ### Подводные камни (для честной оценки)
>
> - **VTab API сложнее, чем кажется.** `xBestIndex` особенно — он определяет, какие `WHERE`-условия передаются в `xFilter`, и неверное планирование приведёт к full-scan вместо индексированного поиска. Без чтения исходников FTS5/R*Tree разобраться сложно — официальная документация формальная и без примеров рассуждения «почему именно так».
> - **SQLite extension API — на C, не C++.** Entry point с `extern "C"`, callbacks как `int (*)(...)` с C-сигнатурами. Ошибки нужно сообщать через `sqlite3_result_error`, а не через C++-исключения (исключение через границу C-API → undefined behavior). Глобальное состояние в extension — opasно, потому что один процесс может загружать extension в несколько `sqlite3*`-handle.
> - **Производительность ниже теоретической.** Чтение `edges` через SQL внутри VTab имеет overhead (binding, type conversion, мультистатус блокировок). На графах < 10K вершин ручная программа на C++ + BGL может опережать. Это нормально, но нужно зафиксировать в отчёте, не скрывать.
> - **Hash map обгоняет любой extension в простом задании «есть ли путь?».** Если задача — connected components, transitive closure достаточно одного раза, не Dijkstra. Включайте такой baseline в сравнения — это интеллектуальная честность.
> - **Готовых extensions нет случайно.** Несколько proof-of-concept-проектов в открытом доступе ограничиваются recursive CTE patterns. Полноценный VTab-extension под графы — это не «велосипед», а реальная ниша, где готового нет. Финальный артефакт уровня 2 имеет смысл выложить на GitHub как open source — потенциально полезный community-проект.
>

#### П28. Повышение точности картирования сложных аритмий

**Что.** Восстановление потенциала миокарда по сигналам всех полюсов баскет-катетера (включая те, что не контактируют со стенкой сердца — на них тоже есть сигнал), для повышения полноты карты активации при сложных аритмиях типа фибрилляции предсердий. Возможные подходы: метод граничных элементов, наименьшие квадраты на CHASTE-моделях, машинное обучение для реконструкции активационных карт.

**Зачем.** Катетерное картирование — приоритетный способ диагностики и терапии аритмий. Баскет-катетеры картируют большую область за раз, но обычная схема теряет данные с электродов в потоке крови; восстановление — критическое улучшение.

- **Минимальный результат:** реконструкция карты активации одним из подходов на синтетических данных из CHASTE.
- **Стек:** C++ + CHASTE + Python (для ML-подхода) + PyTorch.
- **Темы из §0:** Языки программирования · Структуры данных · Распределённые и параллельные вычисления · Нейронные сети. LLM.
- **Трудоёмкость:** R&D.
- **См. также:** П3 (учебная база); М5 (учебный аналог); П29 (родственный домен).

> **Развёрнутое описание (наставник):**
>
> Картирование сердца можно производить как неинвазивно (маверик и wave), так и малоинвазивно с помощью различных катетеров. Катетерное картирование сегодня является самым приоритетным способом диагностики и терапии.
>
> Среди множества катетеров есть такие, которые в той или иной степени достоверности могут картировать сложные аритмии, такие как фибрилляция предсердий. Для этого используются Пента- Окторей катетеры или баскет-катетеры.
>
> Баскет-катетер является перспективным инструментом, поскольку за раз картирует большую область миокарда и позволяет оперативно выявлять области потенциальной аблации. Но тут есть один недостаток: не все контакты баскет-катетера соприкасаются с поверхностью миокарда и, соответственно, карта строится только по тем потенциалам, которые снимаются с электродов, устойчиво контактирующих со стенкой сердца. В результате полнота картины активации утрачивается.
>
> И здесь можно применить подход, аналогичный восстановлению потенциала с тела на сердце — восстановление потенциала миокарда по активности всех полюсов баскет-катетера (даже тех, которые находятся в потоке крови и не контактируют с поверхностью эндокарда — на них тоже есть сигнал, только при обычном способе он не учитывается). Есть два основных подхода как решить эту задачу.
>
> **Первый**, похожий на вейв — когда решается обратная задача методом граничных элементов.
>
> **Второй**, когда за основу берётся CHASTE (<http://www.cs.ox.ac.uk/chaste/>) и решается задача типа наименьших квадратов (подбирается такая картина моделирования, которая даёт наименьшее отклонение от реальной, снятой с полюсов катетера). Второй подход считается неэффективным, поскольку требует много ресурсов для вычислений, но сейчас уже машины мощнее, поэтому можно его попробовать.
>
> В последнее время пробуется **третий** подход — использование машинного обучения для реконструкции активационных карт. Обучающая выборка извлекается из клиник, где проводились соответствующие операции, а также синтезируется с использованием CHASTE. При решении обратной задачи обязательно учитывается характер активности, отражающий движение дипольной цепи, как некоторую модель активационной волны возбуждения миокарда.

**Зачем.** На реальном пациенте не учатся, на рынке таких систем нет, а управление катетером в сердце нетривиально — нужно много часов практики для наработки опыта. Также применимо для планирования операций (виртуальная операция на цифровом двойнике пациента) и академических демонстраций.

- **Минимальный результат:** базовая симуляция картирования на одной модели сердца с управлением мышью/клавиатурой.
- **Стек:** Unity или Unreal + C++ / C# + интеграция с системой картирования Inobitec; опционально — Arduino для ручек, OpenXR для VR.
- **Темы из §0:** Языки программирования · Каркасы программных систем (Frameworks) · Игровые движки · AR/VR разработки · Программные графические интерфейсы.
- **Трудоёмкость:** R&D (3+ лет).
- **См. также:** П28 (источник карт); П1 (источник 3D-модели сердца).

> **Развёрнутое описание (наставник):**
>
> На множестве мониторов симулируем реальность хирургического процесса картирования, ввод и управление катетером, построение поверхностной карты, съём сигнала активности и сам процесс аблации.
>
> Всю физику катетера попробуем моделировать на базе игрового движка — современные компы, думаю, потянут. Построение поверхностной карты возьмём из нашей системы картирования. #### П31. Планирование операций костных восстановлений (виртуальная репозиция)

*Альтернативная формулировка темы:* **Виртуальная репозиция костных фрагментов: алгоритм совмещения на основе ICP.**

**Что.** При тяжёлых травмах кости разрушаются на фрагменты. Хирургу нужно спланировать операцию: как собрать фрагменты обратно. По КТ-снимкам делается 3D-модель, и программа помогает «собрать пазл» — виртуально совместить фрагменты до целостной кости. Алгоритм ICP (Iterative Closest Point) итеративно находит наилучшее совмещение двух наборов точек на краях/поверхностях смежных фрагментов.

**Зачем.** Предоперационное планирование сложных переломов — критическая задача травматологии. Снижает риски и сокращает время операции.

- **Минимальный результат:** ICP-совмещение на синтетических 2D-тестах с валидацией точности.
- **Стек:** Python + NumPy + SciPy + scikit-image + pydicom; визуализация — ParaView.
- **Темы из §0:** Языки программирования (Python) · Структуры данных · Программные графические интерфейсы · Математическое обеспечение.
- **Трудоёмкость:** дипломная / магистерская.
- **См. также:** П14 (родственная задача 3D-математики); П29 (3D-планирование операций).

> **Развёрнутое описание:**
>
> **Алгоритм ICP** (Iterative Closest Point) — итеративно находит наилучшее совмещение двух наборов точек. Применяется к краям/поверхностям смежных фрагментов. Одна итерация: для каждой точки облака A находим ближайшую в B → оптимальное жёсткое преобразование (поворот + сдвиг) через SVD по всем парам → применяем к A → повторяем до стабилизации среднего расстояния. Алгоритм единый для 2D и 3D, меняется только размерность (матрица поворота 2×2 / 3×3).
>
> **Сценарий прототипа:**
>
> 1. Генерация синтетических 2D-тестов: фигура → разбиение на фрагменты → рассеивание (известный поворот+сдвиг каждого).
> 2. Реализация ICP-алгоритма совмещения фрагментов на синтетике.
> 3. Валидация: метрика точности сборки (ответ известен заранее) — среднее расстояние между парами ближайших точек после совмещения.
> 4. Постепенное усложнение: больше фрагментов, шум в координатах, пропущенные куски.
> 5. Переход к 3D: загрузка воксельных масок реальных костных фрагментов (КТ) после сегментации в стороннем ПО (например, Inobitec), извлечение поверхностных точек, ICP-совмещение.
> 6. Сохранение результата как DICOM → визуализация в ParaView / DICOM-просмотрщике.
>
> > Этап синтетических тестов — обязательный: отладка алгоритма на контролируемых данных перед переходом к реальным.
>
> **Расширения:** ограничения непересечения фрагментов (натяг недопустим как нарушение твердотельной сборки); учёт недостающих кусков; метрика качества сборки через оценку зазоров между совмещёнными фрагментами.
