## 4. Правила работы (важно)

### 4.1 Самопроверка перед коммитом

Перед каждым `git commit` / `git push`:

1. `git diff` пофайлово — пробежать глазами всё, что заливается.
2. Проверить: не залип ли мусор (артефакты сборки, временные файлы, чужие правки, незавершённые комментарии).
3. **Прогнать `clang-format -i` на изменённых файлах** — формат должен быть единый, не зависящий от того, в каком редакторе печатала. Должен быть `.clang-format` в корне репозитория (одна стилевая конфигурация для всего проекта).
4. Сборка с `/W4 /analyze` зелёная — без предупреждений, без замечаний встроенного статического анализа.
5. **Release-сборка собирается с `/Zi` + линкер `/DEBUG`** — рядом с exe лежит `.pdb` (Program Database — файл с символами для отладки и анализа дампов). Без pdb любой сбой в эксплуатации программы превращается в неотлаживаемую загадку.
6. Проект собирается с нуля одной командой (на ранних слоях это пакетный сценарий, потом `nmake`, потом `cmake --build`).
7. Тесты проходят (на ранних слоях это вручную запускаемый exe c `assert`-ами, потом `ctest`).
8. Все исправления по последнему ревью внесены.

Это привычка, без которой профессиональная разработка не строится. На встречах наставник смотрит чистоту коммитов — это часть оценки.

### 4.3 STL обязательна на C++-стороне

C++-реализация — **не «C с классами»**, а идиоматический C++. На ревью наставник смотрит именно за этим:

- **Контейнеры** (`std::vector`, `std::array`, `std::string`, `std::map`/`std::unordered_map`) вместо C-массивов, `char[]`, ручного `malloc`.
- **STL-алгоритмы** (`std::sort`, `std::find`, `std::transform`, `std::accumulate`, `std::ranges::*`) вместо ручных циклов с индексом.
- **RAII** — умные указатели (`std::unique_ptr`, `std::shared_ptr`), контейнеры закрывают ресурсы автоматически — вместо ручных init/destroy и `delete`.
- **Range-based for** и **structured bindings** (`for (const auto& [k, v] : map)`) вместо итераторов с индексом.
- **Современные строки и форматирование** — `std::string`, `std::format`, `std::print`/`std::println` (C++23) вместо `char[]` + `sprintf`.
- **Обработка ошибок** — исключения, `std::expected<T, E>` (C++23), `std::optional` вместо кодов возврата + `errno`.

См. шпаргалку «C++ vs C идиомы» — [02-praktika.md](02-praktika.md) § 2.1.B, там карта типичных соответствий.

C-реализация и C++-реализация одной задачи — **разные**, не похожие. C-сторона в этой программе обучения — для понимания, как устроены низкоуровневые механики (управление памятью, errno, ручные буферы). C++-сторона — то, как пишут промышленный код в индустрии.

### 4.4 Параллельная реализация C и C++

**Параллель применяется только на задаче 7.1 (линейные коллекции).** Цель — увидеть и почувствовать разницу подходов на базовом уровне:

- Ручное управление памятью (`malloc/free`) против RAII (умные указатели, контейнеры).
- `errno` и коды возврата против исключений / `std::expected`.
- Индексные циклы против ranges-based for и `std::ranges`.
- `printf` / `fprintf` против `std::cout` / `std::format`.
- `char*` / `strcpy` против `std::string`.

Это образовательный приём, не промышленная практика. Видеть оба подхода важно для C++-разработчика, который пришёл из «C с классами»-стиля. Карта типичных соответствий — в шпаргалке курса наставника, [02-praktika.md](02-praktika.md) § 2.1.B.

На остальных задачах (7.2 B-tree, 7.3 граф, 7.4 линейная алгебра, 7.5 обработка изображений) — сразу C++, без C-параллели.

## 5. Программа подготовки (постепенная)

Модель — **5 базовых задач из § 7, общая инфраструктура которая нарастает по слоям**. Все 5 задач делаются последовательно (одна за другой) в одном репозитории; каждая задача — отдельный подкаталог, но **инструментарий общий** (CMake, `.clang-format`, тесты, непрерывная интеграция). Первая задача создаёт инфраструктурную базу; последующие задачи её используют и расширяют.

Внутри каждой задачи — алгоритмический рост от ядра к более сложным алгоритмам. Параллельно — обрастание слоями инфраструктуры (см. лестницу ниже): на первой задаче подключается базовая сборка из консоли, на второй — CMake, на третьей — тесты через GoogleTest, и т. д.

Лучше **вырастить инфраструктурную базу через все 5 задач** (получить опыт CMake, vcpkg, тестов, непрерывной интеграции на маленьком), потом перенести подход на проект (через магистратуру).

**Стартовая форма любой задачи — параметры зашиты как константы в `main()`.** На первом уровне (L1) каждой задачи из § 7 данные, выбор операции, числовые параметры задаются прямо в коде; программа запускается без аргументов и работает с зашитыми значениями. На L1 фокус — алгоритмическая корректность, без размывания внимания парсингом ввода.

**Лестница слоёв (порядок от низкого к высокому).** Тестовая инфраструктура нарастает рано (слой 8 — простые `assert`-тесты, слой 13 — приёмочные тесты через shell-конвейер, слой 17 — GoogleTest), потому что иметь тесты до подключения сложных оптимизаций и параллелизма — классика снижения рисков ошибок: каждое последующее изменение проверяется автоматически, регрессии ловятся сразу.

### 7.1 Линейные коллекции: поиск + сортировка + параллелизм

**Тематика:** алгоритмическая база, шаблоны, итераторы, STL.

**Параллель C/C++ обязательна** — каждый уровень решается дважды (на чистом C и на идиоматическом C++), чтобы увидеть и почувствовать разницу подходов на базовом уровне (см. § 4.4).

**Постановка задачи.** Программа над коллекцией целых чисел. Три операции:
- **поиск:** индекс первого вхождения заданного значения `K` (или `-1`, если не найдено) и количество сравнений;
- **сортировка:** упорядочение коллекции по возрастанию;
- **статистика:** количество элементов, минимум, максимум, сумма, среднее (с точностью 3 знака после запятой).

**Стартовая форма (L1):** коллекция, выбор операции и значение `K` зашиты как константы прямо в `main()` (например, `std::vector<int> data{5, 2, 8, 1, 9}; int K = 8;`). Программа запускается без аргументов и выводит результат для зашитых данных. Чтение коллекции из стандартного потока ввода и параметры через `argc`/`argv` — отдельные слои, подключаются по лестнице из § 5 (слои 7 и 9). Финальная форма после прохождения этих слоёв: `coll search K < input.txt > output.txt` и т. п.

**Послойный рост:**
- **L1:** свой `MyVector<T>` (или `std::vector`), линейный поиск (`coll search`), вывод исходной последовательности. Подсчёт количества сравнений в выводе `coll search`.
- **L2:** `coll sort` через сортировку выбором или пузырьковую, сравнение времени с `std::sort` через `std::chrono`.
- **L3:** `coll search` через бинарный поиск на отсортированной коллекции (`coll sort` обязателен предварительно).
- **L4:** алгоритмы сортировки слиянием (merge sort), быстрая (quick sort), пирамидальная (heap sort); таблица сравнения теоретической сложности O(N) и реальных замеров.
- **L5:** `coll search` через хеш-таблицу (`std::unordered_map`) — альтернатива бинарному поиску, без необходимости предварительной сортировки.
- **L6:** параллелизм через `std::execution::par` на сортировке и поиске; сравнение замеров с однопоточной версией.

### 7.2 B-tree оптимизация по производительности

**Каноническая постановка — П38** в курсе наставника, [05-projekty.md](05-projekty.md) § 5.1.4. Основан на серии хабр-статей про cache-friendly структуры данных. Имеет 4 уровня глубины — **подходят уровни 1–2** (семестровая + курсовая):

- **L1:** реализация B-tree, операции `insert`/`find`/`delete`, сравнение с `std::set` и `std::unordered_map` на 1 миллион ключей. Хеш-таблица в точном поиске почти всегда обгоняет B-tree (это интеллектуально честно зафиксировать).
- **L2:** B+-tree с linked-list листьями, range-scan, измерение промахов кэша через профилировщик (на Windows — VTune или Visual Studio Profiler), фиксация конфигурации железа в README.

Уровни 3–4 (SIMD-поиск через AVX2, bulk loading, embedded-вариант, каркас для автономного исследования) — опциональны, в магистратуру.

**Параллель C/C++ — нет**, сразу C++.

**Тематика:** память и кэш, профилирование, замеры производительности, эффект иерархии памяти, сравнение с STL-контейнерами на больших массивах ключей.

### 7.3 Поиск минимального пути в графе

**Представление графа:** список смежности (adjacency list), **направленный**, **взвешенный** (чтобы алгоритм Дейкстры имел смысл).

**Постановка задачи.** Поиск кратчайшего пути в графе. На вход — направленный взвешенный граф (вершины 0..N-1, рёбра `src→dst` с весом ≥ 0) и пара `(src, dst)` запроса. На выход — длина кратчайшего пути и сам путь (последовательность вершин через стрелки), либо `INF`, если пути нет.

**Стартовая форма (L1):** граф зашит как константа в `main()` через `std::vector<Edge>` или матрицу смежности; пара `(src, dst)` тоже задана константами. Программа запускается без аргументов и выводит результат для зашитого графа. Чтение графа из файла, запросы из стандартного потока ввода, флаги `--algo=astar` / `--coords=...` / `--unweighted` — отдельные слои, подключаются по лестнице из § 5 (слои 7, 9, 10). Финальная форма: `gpath graph.txt < queries.txt > answers.txt`.

**Формат файла графа** (когда подключится чтение из файла, слой 10):
- Первая строка: `N M` (число вершин и рёбер).
- Следующие `M` строк: `src dst weight` (целые `src`, `dst` от 0 до N-1; вещественный `weight ≥ 0`).
- Запросы из стандартного потока ввода: каждая строка — пара `src dst`.

**Алгоритм по умолчанию — Дейкстра.** При флаге `--algo=astar` и доступном файле с координатами вершин (формат: каждая строка `id x y`) используется алгоритм A* с эвристикой Манхэттена. Флаг `--unweighted` переключает на BFS (Breadth-First Search — поиск в ширину) с длиной пути в количестве рёбер.

**Послойный рост:**
- **L1:** построение графа из файла, обходы BFS / DFS (Depth-First Search — поиск в глубину), поиск пути в невзвешенном графе (флаг `--unweighted`).
- **L2:** алгоритм Дейкстры с `std::priority_queue` на взвешенном графе (поведение по умолчанию).
- **L3:** A* с эвристикой Манхэттена при наличии координат вершин (флаг `--algo=astar`).
- **L4:** оптимизация структур (свой priority queue с операцией `decrease-key`, опционально — Fibonacci heap).

**Параллель C/C++ — нет**, сразу C++.

**Тематика:** алгоритмы на графах, рекурсия и итерация, `std::priority_queue`, эвристики, представления данных.

### 7.4 Линейная алгебра + линейная регрессия

**Постановка задачи.** Программа для матричных операций и регрессии. Две функции:
- **solve:** решение системы `Ax = b` через LU-разложение — на вход матрица `A` и вектор `b`, на выход вектор `x`.
- **regress:** линейная регрессия методом наименьших квадратов — на вход матрица признаков `X` (N строк × p столбцов) и вектор откликов `y` (N значений), на выход коэффициенты `β = (XᵀX)⁻¹Xᵀy` и метрики качества: коэффициент детерминации R² и среднеквадратическая ошибка (MSE — Mean Squared Error).

**Стартовая форма (L1):** матрица `A` и вектор `b` (или `X`, `y`) зашиты как константы в `main()` через `std::vector<std::vector<double>>` или `std::array`. Выбор операции (solve / regress) — компиляцией одного из вариантов. Программа без аргументов выводит результат для зашитых данных. Чтение матриц/векторов из файлов и выбор операции через `argc`/`argv` — отдельные слои по § 5 (слои 7, 10). Финальная форма: `linalg solve A.txt b.txt`.

**Формат файлов матриц/векторов** (когда подключится чтение из файлов, слой 10): первая строка `rows cols`, затем строки с числами через пробелы (для вектора `cols=1`).

**Послойный рост:**
- **L1:** классы `Matrix` и `Vector` (с шаблонами), базовые операции (сложение, вычитание, умножение на скаляр, транспонирование), доступ через `operator[i, j]` (multi-dim subscript C++23). Чтение из файла, вывод. Реализация `linalg solve` для тривиального диагонального случая.
- **L2:** матричное умножение, оптимизация через блочное умножение (cache-blocking): наивное → блочное → сравнение замеров.
- **L3:** LU-разложение, решение `Ax = b` через прямой и обратный ход. Полноценный `linalg solve` для произвольной матрицы.
- **L4:** `linalg regress` через метод наименьших квадратов: `β = (XᵀX)⁻¹Xᵀy`, R², MSE. Численная устойчивость — QR-разложение как альтернатива.
- **L5:** SIMD-оптимизация умножения через встроенные функции компилятора (`__m256` / AVX).

**Параллель C/C++ — нет**, сразу C++.

**Тематика:** численные методы, матричная математика, блочное умножение под кэш, SIMD. Прямая методическая аналогия с твоей ВКР про регрессию маятника, фундамент под медицинскую визуализацию (3D-преобразования, регистрация изображений) — типичную для Инобитека и других производителей медицинского ПО.

### 7.5 Утилита обработки изображений

**Утилита для пиксельной обработки изображений**. Формат — PGM/PPM (простой текстово-бинарный, без сторонних библиотек).

**Постановка задачи.** Программа для пиксельной обработки изображений. Операции:
- **invert** — инверсия яркости.
- **threshold** — бинаризация по порогу `t`.
- **gauss** — фильтр Гаусса с параметром sigma.
- **sobel** — оператор Собеля для выделения границ.

**Стартовая форма (L1):** путь к входному PGM-файлу, путь к выходному файлу, выбор операции и параметры (`t=128`, `sigma=1.5`) — константы в `main()`. Программа запускается без аргументов и обрабатывает зашитые файлы. Чтение через стандартный поток ввода, выбор операции через `argc`/`argv`, флаги — отдельные слои по § 5 (слои 7, 9). Финальная форма: `imgproc gauss --sigma=1.5 < input.pgm > output.pgm`.

**Послойный рост:**
- **L1:** чтение/запись PGM (grayscale) и PPM (RGB), парсинг заголовка, чтение пикселей в `std::vector<uint8_t>`.
- **L2:** пиксельные операции (инверсия, порог, gain, контраст).
- **L3:** свёртка с ядром 3×3 (Sobel для границ, простой Гаусс для размытия).
- **L4:** разделимые фильтры (5×5 Гаусс как два 1×5 прохода), сравнение с naive 5×5 по замерам.
- **L5:** обработка границ (стратегии padding: zero / replicate / reflect).
- **L6:** SIMD-оптимизация свёртки через AVX2.

**Параллель C/C++ — нет**, сразу C++.

**Тематика:** бинарный ввод/вывод, пиксельная обработка, свёрточные ядра, разделимые фильтры, SIMD. Самая прямая аналогия медицинской визуализации (Инобитек и другие производители медицинского ПО).
